# 🛰️ Squadron — 조직 단위 엔지니어링 리소스 매니저

> An organisation-level engineering **resource manager** prototype.
> 50명 이상의 개발자가 3개 이상의 프로젝트를 동시에 진행할 때
> *"지금 누가 비어 있고, 누가 잘 맞고, 무엇이 마감을 막고 있는가"* 를 자동으로 답한다.

가천대학교 자료구조·알고리즘 기말 프로젝트(4인 팀). 각 팀원이 한 모듈을 맡아
**서로 다른 알고리즘 계열**을 구현하고, 하나의 공유 데이터 위에서 조립한다.

---

## 무엇을 하는가

Squadron은 합성 조직 데이터(개발자·프로젝트·스프린트·태스크)를 입력으로 받아 다음을 수행한다.

1. **의존성을 지키는 실행 순서**를 만들고 **임계경로(critical path)** 와 **여유시간(slack)** 을 계산한다 — *Module 1*
2. 스케줄을 따라 **각 태스크를 가장 잘 맞는 개발자에게 자동 배정**한다 (전 일정·용량 인지) — *Module 2*
3. 스프린트 용량 안에서 **가치가 최대인 태스크 묶음**을 고른다 — *Module 3 (예정)*
4. 백로그를 **빠르게 검색·자동완성·오타 보정** 한다 — *Module 4 (예정)*

그리고 이 모든 결과를 **Streamlit 대시보드**로 시각화한다.

## 아키텍처

```
Organisation
└── Projects (3)            P1 Apollo · P2 Borealis · P3 Cirrus  (각기 다른 마감)
    └── Sprints (12)        용량(스토리포인트) 단위
        └── Tasks (153)     의존성 그래프(DAG) · 우선순위 · 필요 스킬 · 가치
Developers (55)             레벨 · 보유 스킬 · 용량/현재부하 · 친숙 프로젝트
```

- **공유 스토어:** 모든 모듈이 같은 JSON 파일들을 메모리(딕셔너리 리스트)로 읽고,
  태스크 그래프의 인접 리스트를 파생한다.
- **모듈 결합도:** Module 1 → Module 2 만 연결되고(아래 *유기적 연동* 참조),
  Module 3·4 는 독립적이라 병렬로 개발 가능하다.

### 모듈 / 업무 분장

각 멤버는 **자료구조 ≥2개, 알고리즘 ≥2개**를 사용하며, 대표 알고리즘은 **서로 다른 계열**이어야 한다.

| # | 모듈 | 대표 알고리즘 (계열) | 보조 알고리즘 | 상태 |
|---|---|---|---|---|
| 1 | 프로젝트 스케줄러 | 우선순위 인지 위상정렬 — 칸(Kahn) + 최소힙 *(그래프)* | 임계경로 DAG 최장경로 DP | ✅ 완료 |
| 2 | 특성기반 배정기 | 헝가리안 / 쿤–먼크레스 *(조합 최적화)* | 그리디 기준선 + trait→affinity 점수화 | ✅ 완료 |
| 3 | 스프린트 플래너 | 0/1 배낭 DP *(동적계획법)* | 백트래킹 + 그리디 밀도 기준선 | ⬜ 예정 |
| 4 | 백로그 검색·자동완성 | KMP / 라빈–카프 + 트라이 *(문자열 매칭)* | 편집거리(퍼지 검색) | ⬜ 예정 |

## 빠른 시작 (Quick Start)

**실행 환경**
- Python **3.8+** (개발·검증은 3.12 에서 수행)
- **채점 대상 모듈(`scheduler.py`, `allocator.py`)은 표준 라이브러리만 사용** — 추가 설치 불필요
- 시각화 앱(`app.py`)만 third-party 의존성 필요 (`requirements.txt`)

```bash
# Module 1 — 스케줄러 (위상정렬 · 임계경로 · 사이클 검출)
python scheduler.py

# Module 2 — 배정기 (전체 일정 자동 배정 + 희소 스킬 데모)
python allocator.py

# 시각화 대시보드
pip install -r requirements.txt
streamlit run app.py
```

## Module 1 — 프로젝트 스케줄러 (`scheduler.py`)

태스크 의존성 그래프를 받아 세 가지를 산출한다.

| 산출물 | 알고리즘 (계열) | 핵심 |
|---|---|---|
| 우선순위 인지 실행 순서 | **칸(Kahn) 위상정렬 + 최소힙** *(그래프)* | 의존성이 *언제* 풀리는지는 진입차수가, *무엇을* 먼저 할지는 힙 키가 결정 |
| 임계경로 + 여유시간 | **DAG 최장경로 DP** *(동적계획법)* | 전진(ES/EF)·후진(LS/LF) 패스, `slack = LS−ES` |
| 달력 타임라인 | 영업일 매핑 | 스프린트 용량에서 velocity 추정 → 마감일에 정렬 |
| 사이클 검출 | **DFS 역간선 복원** *(그래프 탐색)* | 칸 잔여(진입차수>0)로 존재 판정 → DFS로 순환 경로 복원 |

**자료구조:** 인접 리스트(그래프) · 최소힙(`heapq`) · 해시맵(진입차수·ES/EF/LS/LF/slack) · 데크.

**힙 키 (작을수록 먼저):** `(slack, priority, deadline, -out_degree, id)`
임계도(slack=0) → 비즈니스 우선순위 → 이른 마감 → 더 많은 후속을 막는 것 → id(완전 결정성).

**검증된 결과**

```text
P1 (Apollo)  임계경로 89 pts:  T0001 → … → T0007 → T0011 → T0014
P2 (Borealis) 임계경로 56 pts:  T0055 → T0057 → … → T0089
P3 (Cirrus)  임계경로 30 pts:  T0116 → T0119 → … → T0144
```

> 참고: 설계된 무거운 의존성 사슬은 `T0001…T0008` 이지만, 실제 최장경로는
> `T0008` 을 거치지 않고 `T0011 → T0014` 로 빠진다(`T0008` 은 여유가 있음).
> 스케줄러는 README의 기대치에 맞추지 않고 **데이터가 말하는 진짜 임계경로**를 보고한다.

**CLI**

```bash
python scheduler.py                                       # 전체 + 사이클 자가검증
python scheduler.py --project P1                          # P1 상세
python scheduler.py --velocity 4.0 --max-rows 30          # 속도 강제·행수 조절
python scheduler.py --check-only data/tasks_broken.json   # 사이클 검사만
```

`tasks_broken.json` 에는 사이클 1개(`T0001 → … → T0008 → T0001`)가 주입되어 있고,
스케줄러는 이를 검출해 **순환 경로와 스케줄 불가 태스크 22개**를 지목한다.

## Module 2 — 특성기반 배정기 (`allocator.py`)

Module 1이 정한 *"무엇을 먼저"* 를 받아, 각 태스크를 가장 잘 맞는 개발자에게 배정한다.
*"잘 맞는다"* 는 **친화도(affinity)** — 스킬(하드 게이트) + 레벨·난이도 + 프로젝트 친숙도 +
잔여 용량 — 로 정량화한다.

| 알고리즘 (계열) | 역할 |
|---|---|
| **헝가리안 / 쿤–먼크레스** *(조합 최적화, e-maxx O(n²m))* | 전역 최소비용 완전매칭 = 총 친화도 최대화 |
| **그리디 배정** | 비교 기준선 (각 태스크에 그 순간 최적 개발자) |
| **trait→affinity 점수화** | 스킬 게이트 + 레벨/친숙/용량 가중합 |

**자료구조:** 비용 행렬(2-D 리스트) · 집합(스킬 비교 O(1)) · 해시맵(id→레코드, 잠재값, 배정표) · 리스트.

### ⭐ Module 1 ↔ 2 유기적 연동 — 전체 일정 자동 배정

배정기는 단발성 데모가 아니라, **Module 1의 스케줄 전체를 소비**해 백로그 153개를
개발자 55명에게 자동 배정한다.

```
for sprint in scheduler.sprint_order():        # ① Module 1 이 스케줄 순서를 소유(마감 이른 프로젝트 먼저)
    tasks = sprint.tasks (임계경로 먼저)         # ② Module 1 의 slack 이 스프린트 내 우선순위
    while tasks and 가용 개발자:
        round = hungarian(cost[tasks × devs])   # ③ 라운드별 전역 최적 매칭
        commit; remaining_cap[dev] -= estimate  # ④ 용량 = 전 일정 누적 예산(carry-forward)
→ 개발자별: [맡은 태스크], load/가용용량, 가동률
```

- 한 개발자는 잔여 용량이 남는 한 **여러 태스크**를 맡는다(스프린트 내 라운드 누적).
- 마감 이른 프로젝트가 talent 를 먼저 가져간다 → 스케줄의 긴급도가 인력 배분으로 전파.
- **미배정 진단**으로 조직적 신호를 구분한다:
  - **스킬 부재 (16개)** — 필수 스킬 *조합*을 가진 개발자가 0명 → *채용/교육* 신호
  - **전문가 적체 (9개)** — 자격자는 있으나 용량 소진 → *재배치/증원* 신호 (예: `D039` 혼자 5건 병목)

**검증된 결과 (tasks.json 전체)**

| 방식 | 총 친화도 | 배정 | 미배정 | 임계 배정 |
|---|---:|---:|---:|---:|
| Greedy rolling | 9219 | 124 | 29 | 21/28 |
| **Hungarian rolling** | **9546** | **128** | **25** | **21/28** |

헝가리안이 총 친화도 **+327**, 미배정 **29→25** (미배정 25개 = 스킬 부재 16 + 전문가 적체 9).

**희소 스킬 1:1 최소 반례** (왜 전역 최적인가): `rust·spark·terraform·flutter` 는 각 3명만 보유.
그리디는 희소 인력을 평범한 태스크에 먼저 써버려 희소 태스크(`T0091/T0109/T0095`)를 미배정으로
남기고(친화도 234), 헝가리안은 전부 배정한다(460, **+226**).

**CLI**

```bash
python allocator.py                  # ① 전체 일정 자동 배정 + ② 희소 스킬 데모
python allocator.py --full-only      # 전체 일정 배정만(개발자별 워크로드 포함)
python allocator.py --demo-only      # 희소 스킬 1:1 반례만
python allocator.py --sprint P2-S1   # 특정 스프린트 상세 배정
```

## 시각화 대시보드 (`app.py`)

`streamlit run app.py` — 사이드바에서 `tasks.json`(정상 DAG) ↔ `tasks_broken.json`(사이클) 전환.

| 탭 | 내용 |
|---|---|
| 📅 **Module 1 · 스케줄/임계경로** | Gantt 타임라인(작업+여유 막대), slack 표, 우선순위 실행 순서, 의존성 DAG(임계경로 강조) |
| 🔁 **Module 1 · 사이클 검출** | 검출된 순환 경로, 스케줄 불가 집합, 사이클 그래프 |
| 🗺️ **Module 1→2 · 전체 일정 배정** | 조직 집계(그리디 vs 헝가리안), 미배정 진단, 스프린트별 배정 현황, **개발자별 워크로드 + 가동률 막대**, 개발자 드릴다운 |
| 👥 **Module 2 · 배정** | 희소 스킬 반례 / 스프린트 단위 배정, 친화도 히트맵, 이분그래프(태스크→개발자) |

> 사이클이 있는 프로젝트는 임계경로·타임라인이 정의되지 않으므로, 잘못된 수치를 그리지 않고
> 명시적으로 안내한다.
