## 1. 알고리즘 계열 지도

각 멤버는 **자료구조 ≥2 · 알고리즘 ≥2**, 대표 알고리즘은 **서로 다른 계열**이어야 한다.

| # | 모듈 | 대표 알고리즘 (계열) | 보조 알고리즘 | 상태 |
|---|---|---|---|---|
| 1 | 프로젝트 스케줄러 | 우선순위 인지 위상정렬 — 칸 + 최소힙 · **그래프** | 임계경로 DAG 최장경로 DP · DFS 사이클 검출 | ✅ |
| 2 | 특성기반 배정기 | 헝가리안/쿤–먼크레스 · **조합 최적화** | 그리디 배정 · trait→affinity 점수화 | ✅ |
| 3 | 스프린트 플래너 | 0/1 배낭 DP · **동적계획법** | 백트래킹 · 그리디 밀도 | ⬜ 예정 |
| 4 | 백로그 검색·자동완성 | KMP/라빈–카프 + 트라이 · **문자열 매칭** | 편집거리(퍼지) | ⬜ 예정 |

네 대표 알고리즘이 **그래프 / 조합최적화 / 동적계획법 / 문자열매칭** 으로 모두 다른 계열이다.

## 2. 모듈 간 데이터 흐름

```
                          data/*.json  (developers · projects · sprints · tasks)
                                   │
                ┌──────────────────┴───────────────────┐
                ▼                                       ▼
   ┌─────────────────────────┐                ┌──────────────────────────┐
   │  Module 1  Scheduler     │                │  Module 4  Backlog 검색  │
   │  · 위상정렬(Kahn+heap)    │                │  (독립 · standalone)      │
   │  · 임계경로 DP (slack)    │                └──────────────────────────┘
   │  · 사이클 검출            │
   │  · sprint_order()        │
   └───────────┬──────────────┘
               │ slack · critical · sprint_order        (schedule_view)
               ▼
   ┌─────────────────────────┐                ┌──────────────────────────┐
   │  Module 2  Allocator     │                │  Module 3  Sprint 플래너 │
   │  · affinity 점수화        │                │  · 0/1 배낭 DP            │
   │  · 헝가리안 배정          │                │  (Module 1 입력 활용 예정)│
   │  · 전 일정 rolling 배정   │                └──────────────────────────┘
   └─────────────────────────┘
               │
               ▼
        app.py (Streamlit) — 4개 탭으로 시각화
```

**연동 규약:** Module 1만 Module 2·3에 입력을 준다. Module 4는 독립.
현재 빌드된 결합은 **Module 1 → Module 2** 이며, 그 접점은:

- `Scheduler.sprint_order()` — Module 1이 "스케줄 순서"를 소유(마감 이른 프로젝트 먼저).
- `allocator.schedule_view()` — Module 1에서 `slack`·`critical`·`sprint_order`를 추출.
- `Allocator.allocate_schedule()` — 그 순서대로 용량을 소진하며 전 백로그를 배정.

## 3. 복잡도 치트시트

| 모듈 | 연산 | 시간 복잡도 |
|---|---|---|
| 1 | 그래프 구성 | `O(V + E)` |
| 1 | 우선순위 위상정렬 | `O((V+E) log V)` |
| 1 | 임계경로 DP (전진+후진) | `O(V + E)` |
| 1 | 사이클 검출 (DFS) | `O(V + E)` |
| 2 | affinity 1건 | `O(\|skills\|)` |
| 2 | 헝가리안 (1회) | `O(n² m)` |
| 2 | 전 일정 rolling 배정 | 스프린트별 헝가리안 라운드 합 |

`V`=태스크 수, `E`=의존성 간선 수, `n`=배정 태스크 수, `m`=개발자 수.

## 6. 용어집 (Glossary)

| 용어 | 뜻 |
|---|---|
| **DAG** | 방향 비순환 그래프. 태스크 의존성 모델 |
| **위상정렬** | 모든 간선 `u→v`에서 u가 v보다 앞에 오도록 정렬 |
| **ES / EF** | 이른 시작 / 이른 완료 (earliest start/finish) |
| **LS / LF** | 늦은 시작 / 늦은 완료 (latest start/finish) |
| **slack (여유)** | `LS−ES = LF−EF`. 0이면 임계경로 |
| **makespan** | 임계경로 길이 = 프로젝트 최소 완료기간 |
| **임계경로** | slack 0 태스크 사슬. 지연 시 전체 지연 |
| **affinity (친화도)** | 개발자–태스크 적합도 점수(스킬 게이트 + 가중합) |
| **헝가리안** | 최소비용 완전매칭(배정 문제) `O(n²m)` 알고리즘 |
| **velocity** | 영업일당 처리 스토리포인트 |
| **carry-forward** | 개발자 용량을 스프린트 사이에 이월(누적 소진) |
| **stranding** | 그리디가 희소 인력을 잘못 써서 태스크를 미배정으로 남기는 현상 |

## 7. 향후 (Modules 3 · 4)

- **Module 3 — 스프린트 플래너:** 스프린트 용량을 배낭 한도로, 태스크 `value`를 가치로 보는
  **0/1 배낭 DP**. 설계 데모: `P3-S4`(용량 10) — 밀도 그리디는 A(60), 최적은 B+C(80).
  Module 1의 의존성/임계도를 입력으로 활용할 수 있다.
- **Module 4 — 백로그 검색·자동완성:** **KMP/라빈–카프 + 트라이**로 제목·태그 검색과
  자동완성, **편집거리**로 오타 보정. 다른 모듈과 독립.
