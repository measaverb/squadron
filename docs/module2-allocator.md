# Module 2 — 특성기반 배정기: 알고리즘 설명

> 파일: [`allocator.py`](../allocator.py) · 표준 라이브러리만 사용 (Python 3.8+)
> 대표 알고리즘: **헝가리안(쿤–먼크레스) 배정**(조합 최적화)
> 보조 알고리즘: **우선순위 그리디 배정** · **trait→affinity 점수화**
> Module 1 연동: 스케줄러의 임계도(slack)·스프린트 순서로 **전 일정 자동 배정**을 구동

---

## 0. 개요

Module 1이 정한 *"무엇을 먼저"* 를 받아, 각 태스크를 가장 잘 맞는 개발자에게 1:1로 배정한다.
*"잘 맞는다"* 는 **친화도(affinity)** 로 정량화하고, 전체 배정은 **헝가리안**으로 전역 최적화한다.

**핵심 메시지:** 희소 스킬(rust·spark·terraform·flutter; 각 3명만 보유) 보유자는 평범한
태스크에도 매력적이라, 욕심껏 배정하는 그리디는 그 희소 인력을 엉뚱한 곳에 써버려 희소 스킬
태스크를 '미배정'으로 남긴다. 헝가리안은 배정을 **전역 최적**으로 풀어 이를 피한다.

---

## 1. 알고리즘 ① (보조) — trait → affinity 점수화

개발자–태스크 적합도를 하나의 정수로 만든다.

### 1.1 하드 게이트 + 가중합

```text
affinity(dev, task):
    if not (task.required_skills ⊆ dev.skills):   # 하드 게이트: 필수 스킬 미충족
        return None                                # → 배정 불가
    score  = BASE_SCORE(50)
    score += 레벨 적합도   (lvl≥난이도: +12, 과잉 −3/레벨 · 미달 −10/레벨)
    score += 프로젝트 친숙도 (+15)
    score += 잔여 용량 여유 (+10  또는  초과 −4/포인트)
    score += 태그 일치 추가스킬 (+2/개)
    return score
```

| 상수 | 값 | 의미 |
|---|---|---|
| `BASE_SCORE` | 50 | 필수 스킬 충족 기본점 |
| `W_LEVEL_OK` / `W_OVERQUAL` / `W_UNDERQUAL` | 12 / 3 / 10 | 레벨 vs 난이도 |
| `W_FAMILIAR` | 15 | 친숙 프로젝트(램프업 절감) |
| `W_CAP_OK` / `W_OVERLOAD` | 10 / 4 | 잔여 용량 |
| `W_SKILL_SURP` | 2 | 태그 일치 추가 스킬 |

집합(set) 연산 `required ⊆ have`로 스킬 게이트를 `O(1)` 비교한다.

### 1.2 비용으로의 변환

헝가리안은 **비용 최소화**이므로 친화도를 비용으로 뒤집는다.

```text
cost(dev, task) = SCORE_CEIL(200) − affinity        # 친화도 높을수록 비용 낮음
                = BIG(10^7)        if affinity is None   # 배정 불가
```

**계약(contract):** `n × (실현 가능한 최대 실비용) < BIG`. 이 조건이 지켜져야 "총비용 최소화"가
"배정 성공 수 최대화"와 일치한다 — 미배정 1건의 비용 BIG이 실비용 합보다 압도적으로 커서,
헝가리안은 가능한 한 **실현 가능한 배정을 더 많이** 선택하게 된다.

## 2. 알고리즘 ② (대표) — 헝가리안 / 쿤–먼크레스 배정

### 2.1 문제 정의 (Assignment Problem)

이분 그래프(태스크 × 개발자)에서 **각 태스크를 서로 다른 개발자에게** 배정하되
**총비용을 최소화**(= 총 친화도 최대화)하는 **최소비용 완전매칭**.

비용 행렬 `cost[n][m]` (n 태스크 ≤ m 개발자). 개발자가 부족하면 가상 열(비용 BIG)로 패딩한다.

### 2.2 동작 원리 — 잠재값(potential)과 증가경로

e-maxx 구현(`O(n²m)`)을 사용한다. 핵심 불변식은 **상보적 여유(complementary slackness)**:

```
잠재값 u[i](행), v[j](열) 에 대해  reduced(i,j) = cost[i][j] − u[i] − v[j] ≥ 0
매칭 간선은 reduced == 0 (tight) 만 사용
```

행(태스크)을 하나씩 추가하며:
1. 자유 열에서 시작해 **최소 reduced 비용**을 따라 증가경로(augmenting path)를 탐색하고,
2. 경로 상의 모든 라벨을 `delta`만큼 이동시켜(`u += delta`, `v −= delta`) 새 tight 간선을 만들고,
3. 자유 열에 도달하면 `way[]`를 역추적해 매칭을 1개 늘린다.

`n ≤ m`이면 모든 행이 매칭된다. (전제 위반 시 무한루프 대신 `ValueError`로 명확히 실패.)

### 2.3 왜 전역 최적인가 (그리디와의 차이)

그리디는 태스크를 순서대로 보며 "그 순간" 최고 개발자를 집어, 나중 태스크가 쓸 유일 적임자를
먼저 소모할 수 있다. 헝가리안은 **모든 태스크–개발자 짝을 동시에** 고려해 총비용을 최소화하므로,
"한 곳에서 조금 손해 보고 다른 곳에서 크게 이득"인 재배치를 자동으로 찾는다.

### 2.4 희소 스킬 최소 반례 (검증값)

`build_scarce_demo`가 실제 데이터에서 찾아 **검증한** stranding 사례:

| 태스크 | required | GREEDY | HUNGARIAN |
|---|---|---|---|
| T0091 ✦ | csharp,rust | ── 미배정 ── | D035 (65) |
| T0109 ✦ | kubernetes,terraform | ── 미배정 ── | D018 (87) |
| T0095 ✦ | flutter,typescript | ── 미배정 ── | D015 (84) |

```
GREEDY     총 친화도 234 · 미배정 3 · 희소 0/3
HUNGARIAN  총 친화도 460 · 미배정 0 · 희소 3/3      (+226)
```

그리디는 희소 인력을 평범한 태스크 O에 먼저 써버려 희소 태스크 S를 못 채우고, 헝가리안은
O를 대체 개발자에게 주고 희소 인력을 S에 배정해 전부 채운다.

### 2.5 복잡도

$$ O(n^2 m) \quad (n = \text{태스크 수} \le m = \text{개발자 수}) $$

## 3. ⭐ Module 1 ↔ 2 유기적 연동 — 전체 일정 자동 배정 (rolling)

단발성 배정이 아니라 **Module 1의 스케줄 전체**를 소비해 백로그 153개를 개발자 55명에게 자동 배정한다.

### 3.1 Module 1에서 받는 것 — `schedule_view()`

```
slack[t]         태스크별 여유(임계도)               → 스프린트 내 우선순위
critical[t]      임계경로(slack==0) 여부               → '먼저 배정'
sprint_order[]   스프린트 스케줄 순서(마감 이른 순)    → 용량 소진 순서
```

(스케줄러 임포트 실패 시 priority를 임계도 proxy로 쓰는 폴백 경로를 둔다.)

### 3.2 롤링 알고리즘

```text
allocate_schedule(method = hungarian | greedy):
    remaining[dev] = capacity − current_load          # 전 일정 '전역 예산'
    for sprint in sprint_order:                        # ① Module 1 순서대로
        pend = sprint 태스크
        pend.sort( 임계경로 먼저 → slack↑ → priority↑ → id )   # ② Module 1 임계도
        while pend and 가용 개발자 존재:
            round = (hungarian|greedy)(cost[pend × free])      # ③ 라운드별 매칭
            commit feasible(round)                              #    실현 가능분만 커밋
            remaining[dev] -= estimate                          # ④ 용량 소진(이월)
            pend = 못 받은 태스크
        남은 pend = 미배정
```

핵심 설계 3가지:

1. **용량-인지 비용 게이트** `_cap_cost`: 개발자의 잔여 용량 < 태스크 estimate 이면 이번
   라운드 배정 불가(BIG). 이것이 스케줄을 따라 용량을 소진시키는 핵심 제약이다.
2. **스프린트 내 다중 배정**: 라운드는 1태스크/개발자씩 매칭하지만, 용량이 남은 개발자는
   다음 라운드에서 또 맡는다 → 한 개발자가 한 스프린트에서 여러 태스크 누적.
3. **용량 이월(carry-forward)**: `remaining`은 스프린트 사이에 리셋되지 않는다 → 마감 이른
   프로젝트가 talent를 먼저 가져가고, 그 소진이 후속 프로젝트로 전파된다.

### 3.3 왜 '라운드별 헝가리안'인가

한 스프린트에서 "여러 태스크를 용량 한도 내로 한 개발자에게 묶는" 진짜 최적해는
**일반화 배정 문제(GAP, 용량 제약)** 로 NP-hard이다. 라운드마다 1:1 헝가리안(각 라운드는
전역 최적)을 반복하는 방식은, Hungarian 엔진을 그대로 살리면서 용량을 점진 소진하는
**원칙 있는 휴리스틱**이다. 각 라운드 안에서는 여전히 전역 최적 매칭을 보장한다.

### 3.4 미배정 진단 — `classify_unstaffed()`

미배정 태스크를 조직적 의미가 다른 두 부류로 나눈다.

| 부류 | 정의 | 신호 |
|---|---|---|
| **스킬 부재(unstaffable)** | 필수 스킬 *조합* 보유자가 0명 | 채용 / 교육 |
| **전문가 적체(contended)** | 자격자는 있으나 모두 용량 소진 | 재배치 / 증원 |

병목 전문가는 "**시작 시점 가용 용량이 양수**인 자격자"만 집계한다 — 처음부터 과부하였던
개발자를 "재배치하면 풀릴 병목"으로 오인하지 않기 위함이다.

### 3.5 전 일정 검증값 (tasks.json 153 태스크)

| 방식 | 총 친화도 | 배정 | 미배정 | 임계 배정 |
|---|---:|---:|---:|---:|
| Greedy rolling | 9219 | 124 | 29 | 21/28 |
| **Hungarian rolling** | **9546** | **128** | **25** | **21/28** |

미배정 25 = **스킬 부재 16** + **전문가 적체 9**. 병목 전문가는 `D039`가 혼자 5건
(`T0057/T0094/T0064/T0062/T0082`의 유일 자격자이나 **가용 용량 32**(=capacity 38 − load 6)로 과수요).

## 5. 검증 불변식 (전 일정 배정)

`allocate_schedule`는 다음을 만족하도록 검증되었다(11개 불변식 모두 통과):

1. **실현가능성** — 배정된 모든 (태스크, 개발자)가 스킬 게이트 통과
2. **용량 초과 없음** — 개발자별 배정 합 ≤ `capacity − current_load`
3. **전수 1회 배정** — 153개 태스크가 정확히 1행씩
4. **결정성** — 재실행 시 동일 결과
5. **헝가리안 ≥ 그리디** — 총 친화도·배정 수 모두
6. **미배정 정당성** — 미배정 태스크는 자격 개발자가 모두 용량 소진(또는 자격자 0)
7. **스프린트 내 다중 배정 / 프로젝트 간 용량 이월** 실제 발생

추가로 **4-차원 적대적 멀티에이전트 리뷰**(용량 회계·라운드 종료/매칭·Module 1 연동·리포트)
를 통과했다(발견된 minor/nit 4건 모두 수정).
